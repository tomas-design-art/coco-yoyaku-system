## 1. プロジェクト概要

### 1.1 目的

弟が院長を務める接骨院の**紙ボード予約管理を完全にデジタル化**する。

1. 全チャネル（電話・窓口・LINE・HotPepper・チャットボット）の予約を**一元管理**
2. 予約バッティングの**完全防止**（PostgreSQL EXCLUDE制約）
3. HotPepper予約の**自動取り込み**（メール解析）とバッティング検出
4. AIエージェントによる**入力補助・通知・外部連携**
5. 施術中でも対応できる**最小操作のUI**

### 1.2 フェーズ定義（全Phase実装済み）

| Phase | 内容 | 状態 |
|-------|------|------|
| **Phase 1** | 予約台帳UI + CRUD + 競合検出 + 施術者/メニュー/患者管理 + 色分け + 認証 + スケジュール管理 | ✅ 実装済み |
| **Phase 2** | HotPepperメール解析→自動登録、LINE連携、通知システム、HP枠押さえリマインド | ✅ 実装済み |
| **Phase 3** | HPチャットボット予約 + Webチャット予約 | ✅ 実装済み |

**EXCLUDE制約（二重予約防止の心臓部）:**
```sql
CREATE EXTENSION IF NOT EXISTS btree_gist;
ALTER TABLE reservations ADD CONSTRAINT no_overlap
  EXCLUDE USING gist (
    practitioner_id WITH =,
    tstzrange(start_time, end_time) WITH &&
  )
  WHERE (status IN ('CONFIRMED', 'HOLD', 'PENDING'));
```
> アプリ層のバグがあっても、DB層で二重予約を物理的にブロックする。

**source_ref 部分ユニークインデックス（HotPepper二重取込防止）:**
```sql
CREATE UNIQUE INDEX ix_reservations_source_ref_unique
  ON reservations (source_ref)
  WHERE source_ref IS NOT NULL AND status NOT IN ('CANCELLED', 'REJECTED', 'EXPIRED');
```

4. settings の `business_hour_start` / `business_hour_end`（フォールバック）

**施術者勤務判定の優先順位:**
1. `schedule_overrides` にその日のレコードがあるか → あれば最優先
2. `practitioner_schedules` の曜日パターン
3. どちらにもレコードなし → 出勤扱い（営業時間で判定）

### 3.3 予約ステータス

```
PENDING ──────→ CONFIRMED ──────→ CHANGE_REQUESTED ──→ CANCELLED（旧）
   │                │                                     + CONFIRMED（新）
   │                │
   │                └──→ CANCEL_REQUESTED ──→ CANCELLED
   │
   └──→ EXPIRED（HOLD期限切れ）
   └──→ REJECTED（競合で却下）

HOLD ──→ CONFIRMED（承認）
  └──→ EXPIRED（期限切れ）
```

| ステータス | 意味 | 遷移元 | 遷移先 |
|-----------|------|--------|--------|
| **PENDING** | 仮予約 | 新規登録 | CONFIRMED, REJECTED, EXPIRED |
| **HOLD** | 一時確保（時間制限付き） | 変更先枠の仮押さえ | CONFIRMED, EXPIRED |
| **CONFIRMED** | 予約確定 | PENDING, HOLD | CHANGE_REQUESTED, CANCEL_REQUESTED |
| **CHANGE_REQUESTED** | 変更申請中 | CONFIRMED | CANCELLED(旧)+CONFIRMED(新) |
| **CANCEL_REQUESTED** | キャンセル申請中 | CONFIRMED | CANCELLED |
| **CANCELLED** | キャンセル確定 | CANCEL_REQUESTED, CHANGE_REQUESTED | （終端） |
| **REJECTED** | 競合で却下 | PENDING | （終端） |
| **EXPIRED** | 期限切れ | HOLD, PENDING | （終端） |

## 4. 認証・認可

### 4.1 二段階認証方式

| モード | 認証方式 | 対象者 | JWT有効期限 |
|--------|---------|--------|-----------|
| スタッフモード | 4桁PIN（共通） | 全スタッフ | 24時間 |
| 管理者モード | ID + パスワード（bcrypt） | 院長のみ | 30分 |

### 4.2 運用フロー

1. 朝、PCでブラウザを開く → PIN入力画面（テンキー風UI） → スタッフモード
2. 1日中開きっぱなし（24時間セッション）
3. 設定変更が必要なとき → 管理者ログインモーダル → ID+PW → 管理者モード昇格
4. 30分後に自動でスタッフモードに戻る

### 4.3 アクセス権限

|------|:---:|:---:|

| 患者の閲覧・検索・登録・編集 | ✅ | ✅ |
| 通知の閲覧 | ✅ | ✅ |
| HP同期の閲覧・操作 | ✅ | ✅ |
| 施術者の追加・編集・非表示 | ❌ | ✅ |
| メニューの追加・編集・無効化 | ❌ | ✅ |
| 予約色の追加・編集・削除 | ❌ | ✅ |

### 4.4 認証API

| Method | Path | 説明 |
|--------|------|------|
| POST | `/api/auth/staff-login` | PIN認証 → JWTトークン |
| POST | `/api/auth/admin-login` | ID+PW認証 → JWTトークン |
| POST | `/api/auth/logout` | ログアウト |

### 4.5 認証ミドルウェア

- `/api/auth/*` → 認証不要
- `/api/chatbot/*`, `/api/web_chatbot/*`, `/api/web_reserve`, `/api/public/*` → 認証不要（患者向け）
- `/api/line/webhook` → 認証不要（LINE Platform からの呼び出し）
- 管理者権限が必要なエンドポイント → admin JWT 必須
- それ以外 → staff JWT 以上

### 4.6 技術的な注意事項

- `passlib[bcrypt]` + `bcrypt==4.0.1` でピン止め必須（bcrypt 4.2+で`__about__`属性ValueError発生）
- ログアウト→PIN画面遷移: `justLoggedOut` フラグ + LoggedOutScreen で2秒ワンクッション方式

#### 繰り返し予約（シリーズ管理）

- **シリーズ管理導入済み**: `reservation_series` テーブルで繰り返し予約をグループ管理

## 7. ビジネスロジック

### 7.1 自動確定ルール

予約登録時、以下の**全条件**を満たす場合に自動 CONFIRMED:

1. 営業時間内の予約
2. `menu_id` が指定されている
3. `practitioner_id` が指定されている
4. `start_time` と `end_time` が確定している
5. 同一施術者の同時間帯に CONFIRMED/HOLD/PENDING の予約がない
6. 同一患者の同時間帯にダブルブッキングがない

1つでも未充足 → PENDING

### 7.2 予約登録時の検証フロー

```
1. バリデーション
   - start_time < end_time
   - 営業時間内か
   - 5分刻みか
   - practitioner_id が有効か
   - 施術者がその日出勤かどうか（スケジュールチェック）

2. 自動確定判定

3. DB INSERT（EXCLUDE制約でバッティング最終防止）
   - 競合時は IntegrityError → 409 Conflict レスポンス

4. 通知発火（SSEブロードキャスト + notification_log）

5. HotPepper以外のチャネル → HP枠押さえリマインド通知
```

### 7.3 HOLD処理

- 変更申請時の新時間帯を一時確保
- `hold_expires_at = NOW() + hold_duration_minutes` 設定
- APSchedulerが1分間隔で `hold_expires_at < NOW()` を EXPIRED に自動更新
- 失効時にSSE通知

### 7.4 キャンセルフロー

```
キャンセル申請 → status=CANCEL_REQUESTED（枠ロック維持）
  → 通知
  → スタッフ承認 → status=CANCELLED（枠解放）
  → HotPepper経由の場合:「HotPepper側もキャンセルしてください」リマインド
```

### 7.5 変更フロー

```
変更申請（新日時指定）→ 新時間帯をHOLDで確保
  → 旧予約: CHANGE_REQUESTED
  → 通知（旧+新を並べて表示）
  → スタッフ承認 → 旧: CANCELLED, 新HOLD: CONFIRMED
  → HotPepper経由の場合:「HotPepper側も変更してください」リマインド
```

### 7.6 リスケジュール

- 予約の日時・施術者変更を1アクション（HOLD/CHANGE_REQUESTED経由なし）で実行
- 管理者/スタッフが直接操作

### 7.7 突発休み時の振替候補提示

```
管理者が臨時休み登録 → バックエンドがその日の既存予約を自動チェック
  → 影響予約あり: affected_reservations + transfer_candidates を返却
  → フロントが振替モーダル表示
  → 各予約について:
     [振替する] → 旧CANCELLED + 新CONFIRMED
     [別日に変更] → 変更申請フロー
     [キャンセル] → キャンセル申請フロー
     [そのまま] → 何もしない
```

### 7.8 繰り返し予約（シリーズ管理）

- 頻度: 毎週 / 隔週 / 毎月
- 終了条件: 終了日 or 回数（最大13回 ≒ 約3か月）
- 個別生成 → 競合/休診/不在はスキップ
- 結果: 作成件数 + スキップ理由のリスト + `series_id`
- **シリーズテーブル** (`reservation_series`):
  - `patient_id`, `practitioner_id`, `menu_id`, `color_id`
  - `start_time` (HH:MM), `duration_minutes`, `frequency`
  - `remaining_count`, `total_created`, `notified_at`, `is_active`
- **予約リンク**: `reservations.series_id` FK → `reservation_series.id`
- **残り3回通知**: 毎日9:00 JST、`remaining_count ≤ 3` かつ `notified_at IS NULL` → `series_expiring` 通知生成
- **延長**: 同設定で追加（最大13回）、`notified_at` リセット
- **変更**: 未来予約キャンセル → 新設定で再生成
- **キャンセル**: 未来予約を全CANCELLED、シリーズ非アクティブ化

### 7.9 定期ジョブ（APScheduler）

| ジョブ | 間隔 | 内容 |
|-------|------|------|
| `expire_holds` | 1分 | HOLD期限切れ → EXPIRED |
| `expire_chat_sessions` | 10分 | チャットセッション24時間超 → expired |
| `remind_hotpepper_sync` | 30分 | HP未同期予約のリマインド通知 |
| `check_series_expiration` | 毎日9:00 | シリーズ残り3回以下 → series_expiring通知 |
| `cleanup_old_notifications` | 毎日3:00 | 古い通知ログの自動削除 |

### 7.10 スマートスロットスコアリング

空き枠提案時の最適候補スコアリング:
- ゴールデン枠（人気時間帯）ボーナス
- 空白ペナルティ（隙間なく詰めるほうが高スコア）
- スタッフバランス（偏りを防ぐ）

## 8. 外部チャネル連携（Phase 2）

### 8.1 HotPepperメール解析

#### フロー

```
IMAPメール取得（IMAP4_SSL / iCloud対応）
  → 件名フィルタ（HotPepper予約通知）
  → ルールベースパーサー（正規表現で構造化抽出）
  → AI review/patch（Gemini — 欠損フィールドの補完）
  → 必須フィールド検証（名前/日時/時間/施術者希望）
  → 予約登録（channel=HOTPEPPER, source_ref=予約番号）
```

#### 対応メールパターン

| # | 件名パターン | event_type | 処理 |
|---|-------------|-----------|------|
| 0 | 予約連絡 | `created` | 予約登録 |
| ① | 【明日】キャンセル連絡 | `cancelled` | 予約キャンセル |
| ② | 直前予約が入りました | `created` | 予約登録 |
| ③ | 【明日】予約連絡 | `created` | 予約登録 |
| ④ | 未対応予約のお知らせ | `reminder` | 処理対象外（ValueError） |

#### 抽出フィールド

reservation_number, patient_name, start_time, end_time, duration_minutes, practitioner_name, menu_name, amount, coupon_name, note, received_at

#### HotPepper特有ルール

- HotPepper予約は**競合があっても登録する**（外部で確定済みのため）
- 競合検出時は `conflict_note` に記録 + 強アラート通知
- `source_ref` の部分ユニークインデックスで二重取込防止
- 必須フィールドが揃わない場合は「ホットペッパー予約者のシステム反映がされていません」通知を発出

#### ポーリング

- APScheduler で定期実行（間隔は設定可能）
- 手動実行ボタン (`POST /api/hotpepper/trigger-poll`)
- 稼働状態確認エンドポイント (`GET /api/hotpepper/runtime-status`)

### 8.2 HotPepper枠押さえリマインド

```
本システムに予約登録（電話/窓口/LINE/チャットボット）
  → 通知:「HotPepper側の {date} {time} を押さえてください」
  → /hotpepper 画面で未押さえ一覧確認
  → 「押さえ済み」ボタン → hotpepper_synced = true
  → 未押さえ予約は30分間隔でリマインド通知
```

### 8.3 LINE連携

#### Webhook受信

LINE公式アカウントの Webhook (`POST /api/line/webhook`) で受信。

#### 動作モード

| モード | 設定値 | 動作 |
|--------|--------|------|
| シャドーモード | `line_mode=shadow` | 受信メッセージをAI解析のみ。自動返信なし。管理者に解析結果を通知 |
| アクティブモード | `line_mode=active` | AI秘書として自動返信・予約受付 |

## 10. LINE AI秘書

### 10.1 概要

LINE公式アカウントと連携し、患者からのLINEメッセージをAIが解析・返答する。

### 10.2 対話状態管理

`line_user_states` テーブルで対話状態をDB永続化:
- `line_user_id`: LINEユーザーID
- `current_step`: 対話ステップ
- `context_data`: JSONB（収集中の予約情報等）

### 10.3 プライバシー保護

- 来院歴確認 → 電話番号 or 生年月日で本人確認 → 既存患者リンク
- 新規利用者 or 確認失敗 → 新規患者登録として対応
- 名前は区切りあり → split / 区切りなし → full_name モードで保存

### 10.4 シャドーモード

- `line_mode=shadow` 時: 受信メッセージをAI解析のみ（自動返信無し）
- 解析結果を `shadow_logs` テーブルに記録
- 管理者に通知して確認してもらう
- `/api/shadow-logs/` で解析ログ閲覧可能

### 10.5 予約フロー（アクティブモード）

```
患者がLINEメッセージ送信
  → Webhook → AI解析（予約意図あり/なし）
  → 予約情報収集（日時/メニュー/名前等）
  → スマートスロットスコアリングで最適候補提示
  → 仮予約 → 管理者承認待ち通知
  → 管理者が承認 → CONFIRMED → LINEで確認メッセージ送信
```

### 10.6 LINE通知

- `line_alerts.py`: 管理者への予約レビュー通知（Flex Message）
- `line_reply.py`: 患者へのメッセージ返信（テキスト/クイックリプライ/Flex Message）
- SOS通知: 障害発生時に開発者へ自動アラート

## 11. 非機能要件

### 11.1 パフォーマンス

- タイムテーブル表示: 1秒以内
- 予約登録: 500ms以内
- SSE通知: 2秒以内に全クライアントに配信

### 11.2 セキュリティ

- 二段階認証（PIN + 管理者ID/PW）
- JWT（HS256署名、`SECRET_KEY` 環境変数）
- 管理者パスワードはbcryptハッシュで保存
- APIはCORSで許可ドメインのみ
- チャットボットにレート制限
- 入力サニタイズ
- SQLインジェクション対策（SQLAlchemy ORM）

### 11.3 可用性

- Docker Compose で一括起動（開発・本番）
- PostgreSQL のバックアップ設定
- `.env` で環境変数管理（APIキー等をコードに直書きしない）

### 11.4 デプロイ

- 開発: `docker-compose.yml`（localhost:5173 + localhost:8000）
- 本番: `docker-compose.prod.yml` + nginx（SSL対応）
- 手順: `DEPLOY.md` に記載

### 11.5 テスト

- バックエンドテスト: pytest（12テストファイル、200+テストケース）
- テスト対象: 競合検出、祝日、HotPepper解析、ポーリング、患者インポート、予約CRUD、シャドーモード等

### 11.6 日時管理

- すべてJST（`Asia/Tokyo`）で統一
- バックエンド: `utils/datetime_jst.py` → `now_jst()` / `JST = ZoneInfo("Asia/Tokyo")`
- フロントエンド: `timeUtils.ts` にJST算術（UTC+9h）
- DB: TIMESTAMPTZ（timezone-aware）

## 12. 設計上の重要な決定事項

| 決定 | 理由 |
|------|------|
| React採用（Streamlit不採用） | ドラッグ操作・5分刻みUI・音通知・SSEはStreamlitでは不可能 |
| PostgreSQL必須 | EXCLUDE制約による二重予約防止。SQLiteでは不可能 |
| HotPepper予約は競合があっても登録 | 外部で確定済みのため拒否すると整合性破壊。競合アラートで人間が判断 |
| キャンセル・変更は人間承認制 | AIは提案まで。現場判断が必要な操作は人間が最終決定 |
| async SQLAlchemy で commit 後は selectinload 再クエリ | `db.refresh(obj, [rels])` はスカラー属性を expired のまま残し `MissingGreenlet` を引き起こす |
| 繰り返し予約はシリーズ管理導入済み | `reservation_series` テーブルで最大3か月管理、残り3回通知→延長/変更/キャンセル |
| LINE秘書はシャドーモードから開始 | 本番環境で精度検証してからアクティブモードに切り替え |
| HotPepper解析はルールベース+AI review | メール構造が定型のためルールベース主体、AI は補完のみ |
| 施術者削除は2段階 | 論理削除 → 予約なし確認 → 物理削除（データ保全） |
| 名前登録は2モード | 区切りあり → split（姓名分割）, 区切りなし → full_name（フル入力） |
| bcrypt 4.0.1 固定 | 4.2+で passlib の `__about__` 属性参照がValueError |
| FastAPI URLの末尾スラッシュ | `client.ts` のリスト取得・POST URLに必ず末尾 `/` を付ける |
